Broad strokes

Web Application Design Concerns

Some high-level themes to think about...
  • Data Sourcing and Storage
  • API and Service Layers
  • User Interface and Visualization
  • This assumes a 3-part architecture that separates data storage, business logic, and interface.
  • API is Application Programming Interface. An API is tool used as the business layer/logic layer to access your data. It's the thing power users and 3rd-party apps will use.
  • RESTful - Representational State Transfer. REST is an architectural pattern for creating web services using HTTP. stateless API- server does not maintain state. just responds to requests.
  • User interface: contact pont between users/audience and the science/data. Many people will only see it in this form.

Data Sourcing and Storage

  • Dynamic (real-time) will be a guiding principle for IWAAs.
  • Real-time (or near real-time) requires some combination of automated processes, data harmonization and a database storing the output of both.
  • the product we eventually make will be a real-time (or near real-time) product
  • 'real-time' has many defintions
  • "deliver a national real-time census of water resources, integrating water supply, use, and influencing factors"

Data Sourcing and Storage

Process development and data harmonization should be designed with the goal in mind: a performant database and API.
Standardizing the language, processes, and outputs will be key.

API and Service Layers

An API has two major roles:
  • Serve the application.
  • Serve the power users and 3rd-party applications.


Uniting these roles as much as possible will bring efficiencies.
  • A well documented API has its own web presence.
  • If you don't have a documented API, you don't really have an API

Data Sourcing and Storage

Bad performance in a web application will undermine the message.
API users have more patience. Typically where the tougher requests are satisfied.
  • Web application should optimize a positive user experience
  • Anyone remember rollout of Healthcare.gov?

API and Service Layers

Map services

Map services are more specialized to map visualizations

You don't need a map service to have a map. May or may not be the right approach for the visualization.
  • Esri ArcGIS Server most common within USGS
  • Geoserver is an open-source option.
  • Can also serve map tiles - but these will need to be generated for each data update.

User Interface and Visualization

User interface should be intuitive and friendly
Client-side rendering vs. Server-side rendering
  • Map services, image services, tile services allow for a server-side implementation
  • Much can be rendered on the client with simple-format data served from the API
  • Performance considerations in either direction. Balance to strike.
